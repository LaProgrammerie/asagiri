# Agenten-Übersicht

Asagiri ist eine Orchestratorschicht: Es werden keine Modellgewichte mitgeliefert und kein Hersteller-Zugang „aus der Box“. Sie deklarieren **externe CLIs** unter `agents:`; Routing, Standardwerte in `work` oder Befehlszeilen-Flags entscheiden, welches Executable für einen Schritt startet.

## Unterstützte Muster

Die Tabelle zeigt gängige logische IDs. Welches Binary tatsächlich läuft (`codex`, `cursor-agent`, Wrapper …) und welche statischen Argumente gesetzt werden, steht ausschließlich in **Ihrer** `config.yaml`. Asagiri startet `command` ohne Shell mit optionalem `argv` und leitet Streams über den bestehenden Redaktionspfad für Geheimnisse.

| Agent-ID (Beispiel) | Typische Rolle |
| --- | --- |
| `kiro` | Verfassen strukturierter Specs (`asa spec`) |
| `cursor` | Umsetzung (`dev`, häufiger Standard für `work`) |
| `codex` / `claude` | Review oder intensiveres Reasoning |
| `ollama` | Lokales Anreichern, Klassifikation, günstiges Pre-Review |

## Verträge zwischen CLI und Agent

Die Exec-Schicht arbeitet ohne Shell-Interpolation und nur mit explizitem `argv`. Timeouts kommen aus `agents.<id>.timeout`. Unter `--dry-run` werden keine produktiven Subprozesse gestartet. Logausgaben durchlaufen die Redaktion in `application/internal/redact`.

## Agenten pro Schritt wählen

Wechseln Sie die Agent-ID pro Subbefehl, wenn sich die passende Werkbank zwischen Spezifikation, Implementierung und Review unterscheidet.

```bash
asa spec feat --agent kiro
asa enrich feat --agent ollama
asa dev feat --agent cursor
asa review feat --agent codex
```
