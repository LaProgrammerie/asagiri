# Mitwirken

Checkliste, Review-Erwartungen und Repository-Richtlinien stehen im **[CONTRIBUTING.md](https://github.com/LaProgrammerie/asagiri/blob/main/CONTRIBUTING.md)** im Repo-Stamm — dort beginnen Sie bei allem, was über diese Kurzübersicht hinausgeht.

## Kurz-Checkliste vor einem Pull Request

Projektkontext laden, Umfang mit aktivem Handoff und Produktspecs abstimmen, nachweisen dass der Go-Baum baut und die Tests grün sind, bei CLI-Änderungen die Referenz neu erzeugen, bei MDX-Änderungen den Docs-Site-Build fahren und jeden PR auf ein zusammenhängendes Thema begrenzen — ohne Geheimnisse in Commits.

1. `AGENTS.md` und `docs/ai/context-map.md` lesen
2. Features an `docs/ai/active/handoff.md` und die Produktspecs (z. B. `spec-postv123.md`) ausrichten
3. `make build && go test -race ./...`
4. Bei Änderungen an Befehlen die CLI-Doku aktualisieren:

```bash
go run ./application/cmd/asa docs generate-cli \
  --output docs-site/content/docs/en/cli/generated
```

5. Bei MDX-Änderungen: `cd docs-site && npm run build`
6. Ein PR pro kohärentem Thema; keine Secrets committen

## Dokumentation

Öffentlicher Fließtext liegt unter `docs-site/content/docs/` (Fumadocs für die englische Site). Interner Kanon bleibt unter `docs/ai/`. Wo Code und Doku auseinanderlaufen, hilft [`problems.md`](https://github.com/LaProgrammerie/asagiri/blob/main/problems.md) beim Tracking.

## Architekturentscheidungen

Dauerhafte Trade-offs dokumentieren Sie dort, wo spätere Mitwirkende suchen: `docs/ai/05-decisions.md` und bei Bedarf strukturierte Einträge unter `docs/decisions/`.
