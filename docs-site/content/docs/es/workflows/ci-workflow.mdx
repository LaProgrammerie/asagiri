# Flujo CI

La integración continua debe ejercitar Asagiri **sin prompts interactivos** — preferible modo **dry-run** cuando ese matiz coincide con su intención — y sin volcar secretos al log público CI.

## Patrón sugerido

Un carril típico compila CLI, inicializa proyecto, ejecuta `doctor`, luego pasa **`work` dry-run + plan-only** para mantener rápido y determinístico.

```yaml
# extracto — adapte plataforma
- run: go build -o bin/asa ./application/cmd/asa
- run: bin/asa init
  working-directory: ${{ github.workspace }}
- run: bin/asa doctor
- run: bin/asa work "smoke feature" --dry-run --plan-only
- run: go test -race ./...
```

## Variables entorno útiles

Preferir variables frente repetir switches individuales:

| Variable | Efecto |
| --- | --- |
| `ASA_DRY_RUN=1` | Equivalente `--dry-run` global |
| `ASA_YES=1` | Auto confirma prompts (solo consciente CI) |
| `NOTION_TOKEN` | Obligatorio rutas sincronización Notion |

Véase [Variables de entorno](/docs/es/reference/environment-variables).

## Docs CI

Validación también abarca compilación site publicado + regenerate CLI inglés coincide código:

```bash
cd docs-site && npm ci && npm run build
go run ./application/cmd/asa docs generate-cli --output docs-site/content/docs/en/cli/generated
go test ./application/internal/cli/docgen/...
```

Definiciones `.github/workflows/docs.yml`.

## Qué omitir conscientemente CI

Ci es ensayo—not production automation unattended unless pinning agents scopes secrets audited:

- no correr review/dev real sin pines
- nunca configuración con tokens versionada
- no sustituir toda seguridad sólo métricas estimación tokens
