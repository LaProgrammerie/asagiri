# Workflow soucieux des coûts

Cette page opérationnalise **`spec-doc` §11.4** : chaque drapeau et clé listés existent bien dans vos builds Asagiri et enchaîne estimation, optimisation des packs budgétés, puis préférences **locales** là où votre stratégie de routage permet — commandes verbatim reproduites pour écarter ambiguïtés.

## 1. Estimation par tâche

```bash
asa estimate billing-v2 --task task-003 --dry-run
```

Affiche entrées/sorties heuristiques + coût estimé suivant tableau `pricing`.

## 2. Optimisation contextual

```bash
asa context billing-v2 --task task-003 --optimize
```

Construire pack avant gros prompts. Désactiver avec `asa work ... --no-context-reduction`.

## 3. Lancer sous budget

```bash
asa work "develop billing-v2" --budget 0.50 --estimate-only
asa work "develop billing-v2" --budget 0.50
```

`budgets.per_run.max_estimated_cost` s’active quand `--budget` vaut zero. Bloque si surcharge sauf confirmations prompt adoptées **`--yes` après BudgetPending**, ou **`--allow-over-budget`** autorisée par policies config.

## 4. Priorité locale

```bash
asa work "develop billing-v2" --prefer-local --no-cloud
```

`--allow-cloud` ré‑admet cloud où restrictions le demandent.

## Explainabilité

`work` + `estimate` impriment jalons synthétiques : choix routing, octets contextual, validations budget voir aussi [Estimation jetons](/docs/fr/cost-performance/token-estimation).
