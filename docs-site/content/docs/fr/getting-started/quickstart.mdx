# Démarrage rapide

Ce parcours utilise **`--dry-run`** pour explorer l'orchestration sans invoquer d'agents externes ni de shells de validation.

## 1. Compiler et initialiser

```bash
make build
export PATH="$PWD/bin:$PATH"
cd your-repo
agentflow init
agentflow doctor
```

## 2. Dry-run d'un workflow en langage naturel

```bash
agentflow work "develop my-feature" --dry-run --plan-only
agentflow work "develop my-feature" --dry-run --estimate-only
```

`--plan-only` résout l'intention et affiche le plan d'exécution. `--estimate-only` exécute le pipeline V3 jusqu'à l'estimation coût/contexte sans exécution agent.

## 3. Inspecter les commandes du pipeline

Avec un dossier de spec Kiro existant (par ex. `.kiro/specs/my-feature/`) :

```bash
agentflow plan my-feature --dry-run
agentflow enrich my-feature --dry-run
agentflow dev my-feature --dry-run
agentflow verify my-feature --dry-run
agentflow review my-feature --dry-run
agentflow status --dry-run
```

## 4. Rapports

Après un `work` en dry-run qui crée les métadonnées d'exécution :

```bash
agentflow status --dry-run
agentflow report <run-id> --dry-run
```

Les rapports sont écrits sous `.agentflow/runs/<run-id>/report.md`.

## Exécution réelle

Retirez `--dry-run` uniquement lorsque les agents et les commandes de validation sont configurés et que vous acceptez l'exécution de processus externes :

```bash
agentflow work "develop my-feature" --estimate-only   # garde coût d'abord
agentflow work "develop my-feature"                   # confirmation guidée par défaut
```

<Callout type="warning">
Le dry-run simule l'orchestration ; il ne garantit pas la qualité de sortie des agents. Exécutez toujours la validation du projet (`go test`, linters, revue humaine) sur les changements réels.
</Callout>
