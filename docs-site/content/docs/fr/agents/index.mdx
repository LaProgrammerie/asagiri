# Vue d'ensemble des agents

Asagiri est une couche d’orchestration : il n’embarque ni poids de modèles ni accès propriétaire. Vous déclarez des **CLI externes** sous `agents:` ; le routage, les valeurs par défaut de `work` ou les drapeaux par commande déterminent quel binaire s’exécute pour une étape donnée.

## Modèles courants

Le tableau liste des identifiants fréquents ; l’exécutable et les drapeaux restent **votre** `config.yaml` — le contrat d’Asagiri est de lancer `command` avec des `args` optionnels et de faire transiter les flux via le chemin de rédaction habituel.

| Identifiant d’agent (exemple) | Rôle typique |
| --- | --- |
| `kiro` | Rédaction de spec (`asa spec`) |
| `cursor` | Implémentation (`dev`, agent par défaut de `work`) |
| `codex` / `claude` | Revue ou raisonnement lourd |
| `ollama` | Enrichissement local, classification, pré-revue économique |

## Contrats

- Pas d’interpolation shell — uniquement `argv`
- Délais d’attente via `agents.<id>.timeout`
- Le dry-run évite l’exécution réelle
- Les journaux passent par la rédaction (`application/internal/redact`)

## Choisir les agents par étape

Changez l’identifiant d’agent par sous-commande lorsqu’une tâche doit s’appuyer sur une autre chaîne d’outils que vos défauts.

```bash
asa spec feat --agent kiro
asa enrich feat --agent ollama
asa dev feat --agent cursor
asa review feat --agent codex
```
