## When to use

Workflow commands advance tracked features across spec → plan → implementation → verification → reporting guardrails before opening a merge request.

## Usage

```bash
asa verify trust [flags]
```

## Options

| Flag | Scope | Default | Description |
| --- | --- | --- | --- |
| `--branch` | Local | `(empty)` | Branche git ciblée |
| `--ci` | Local | `false` | Code de sortie non nul si gate bloqué ou checks en échec |
| `--dry-run` | Inherited | `false` | Simuler les exécutions sans lancer d'agent ni commandes externes |
| `--flow` | Local | `(empty)` | ID de flow (remplace l'argument positionnel) |
| `--json` | Local | `false` | Sortie JSON du trust report sur stdout |
| `--product` | Local | `(empty)` | ID produit sous .asagiri/products/ |
| `--strict` | Local | `false` | Traiter les checks en warn comme des échecs en mode CI |
| `--task` | Local | `(empty)` | ID de tâche associée |
