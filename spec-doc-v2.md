# Mission — Editorial rewrite & documentation humanization pass
Objectif :
Réaliser une passe éditoriale complète sur la documentation AgentFlow afin de transformer une documentation actuellement trop “bullet points / autogenerated reference” en une documentation d’ingénierie claire, fluide, pédagogique et agréable à lire.
IMPORTANT :
Le but N’EST PAS de changer le fond technique.
Le but est d’améliorer :
- le ton ;
- la fluidité ;
- les transitions ;
- les explications ;
- la pédagogie ;
- le confort de lecture ;
- la cohérence éditoriale ;
- la crédibilité technique.
La documentation doit ressembler à :
- une documentation produit d’ingénierie premium ;
- une documentation écrite par des humains expérimentés ;
- une documentation fiable et réfléchie ;
- pas à des fragments générés automatiquement.
Inspiration :
- Kiro CLI docs
- Vercel docs
- Cloudflare docs
- Terraform docs
- Stripe engineering docs
---
# Contraintes ABSOLUES
## 1. NE PAS casser le contenu technique
Ne pas :
- supprimer des concepts ;
- inventer des fonctionnalités ;
- modifier le comportement documenté ;
- réécrire des exemples en changeant leur sens ;
- simplifier à outrance.
Le travail est éditorial et pédagogique, pas fonctionnel.
---
## 2. NE PAS oublier les traductions
IMPORTANT :
Toute modification d’une page doit être répercutée dans toutes les langues existantes.
Si une page existe :
- en anglais ;
- en français ;
- ou dans d’autres langues,
alors :
- le ton ;
- les transitions ;
- la fluidité ;
- les explications ;
- la structure narrative
doivent être cohérents dans toutes les versions.
Interdiction :
- d’améliorer uniquement la version anglaise ;
- de laisser une traduction plus “robotique” ;
- de créer des divergences éditoriales importantes entre langues.
Les contenus traduits doivent paraître écrits naturellement dans leur langue respective.

# Problème actuel
La documentation actuelle :
- est techniquement correcte ;
- est bien structurée ;
- possède une bonne navigation ;
- possède un bon design ;
mais le ton est souvent trop :
- fragmenté ;
- télégraphique ;
- “liste de features” ;
- “CLI reference générée automatiquement”.
Exemples problématiques :
BAD:
```md
Cost and token estimation — see spend before you run.

BAD:

Local investigation
Bounded grep and filesystem traversal before cloud calls.

BAD:

Agent routing — pick the right tool per step.

Ces formulations sont :

* trop courtes ;
* peu pédagogiques ;
* peu naturelles ;
* peu éditorialisées ;
* sans contexte ;
* sans explication ;
* trop proches d’un catalogue technique.

⸻

Ce qu’on veut à la place

GOOD:

AgentFlow estimates token usage and execution cost before a workflow starts.
Instead of blindly sending large amounts of repository context to cloud models, the system first performs local investigation and context reduction. This makes workflows more predictable and helps developers understand the expected cost before execution begins.

GOOD:

Local investigation happens before any cloud model execution.
Rather than relying on large prompts containing entire repositories, AgentFlow scans the project locally using bounded filesystem traversal, grep, dependency analysis, and relevance heuristics in order to reduce the amount of context sent to external models.

Les nouvelles pages doivent :

* expliquer ;
* contextualiser ;
* raconter légèrement ;
* introduire les concepts ;
* expliciter les choix techniques ;
* décrire les compromis ;
* inspirer confiance.

⸻

Style guide obligatoire

1. Préférer la prose explicative

Chaque section importante doit contenir :

* de vrais paragraphes ;
* des transitions ;
* des explications ;
* des exemples ;
* du contexte.

Une page ne doit pas ressembler à :

* une checklist ;
* un dump de références CLI ;
* une liste de slogans.

⸻

2. Expliquer WHY + HOW

Chaque page doit expliquer :

* le problème d’ingénierie ;
* pourquoi AgentFlow traite ce problème ;
* comment le système fonctionne ;
* les bénéfices ;
* les limites ;
* les trade-offs.

⸻

3. Réduire les bullet points

Les bullet points doivent :

* compléter la prose ;
* pas remplacer l’explication.

Ratio cible :

* ~70% prose explicative
* ~30% listes / références / tableaux

⸻

4. Ajouter des transitions naturelles

Éviter :

CLI
Configuration
Trade-offs

Préférer :

Once context optimization is enabled, several commands become available for inspecting and debugging generated context packs.

⸻

5. Ajouter des exemples concrets

Mauvais :

Agent routing selects models intelligently.

Meilleur :

A small refactoring task may run entirely on a local Ollama model, while a large migration involving many files can automatically escalate to a cloud model with a larger context window.

⸻

6. Documenter honnêtement les limites

Toujours expliquer :

* ce que le système améliore ;
* ce qu’il ne garantit pas ;
* où les heuristiques peuvent échouer ;
* quand la revue humaine reste nécessaire.

⸻

7. Éviter le ton “AI magic”

Interdiction des formulations :

* autonomous intelligence
* self-healing AI
* fully autonomous coding
* magic AI
* smart AI orchestration

Préférer :

* deterministic workflows
* explicit routing
* validation-driven workflows
* reproducible execution
* measurable cost estimation

⸻

8. Le ton doit inspirer

Le lecteur doit ressentir :

* contrôle ;
* fiabilité ;
* explicabilité ;
* transparence ;
* engineering sérieux ;
* maîtrise des coûts ;
* reproductibilité.

Pas :

* hype ;
* magie ;
* marketing LLM.

⸻
